# FOUNDERLAB AI — PERMANENT ENGINEERING RULES

FounderLab already exists. NEVER rebuild existing tools. Always improve and extend them without breaking working functionality.
Every task must reach production quality before being considered complete. "Works" is NOT enough. It must feel premium.
Never use templates, placeholder content, fake brands, lorem ipsum, old layouts, outdated UI, or generic boilerplate.
Never generate websites, UIs, or code that look outdated. Build for modern 2026 SaaS quality.
Every feature must look comparable to the best modern products (Linear, Vercel, Framer, Raycast, Notion, Stripe, Cursor) while keeping FounderLab's own identity.
Every button, navigation item, animation, preview, form, modal, shortcut and workflow must actually work. Never add fake buttons.
Never stop after implementing a feature. Review your own work, find weaknesses, improve them, then deliver.
If something looks average, redesign it before finishing.
Every generated website must be unique to the user's prompt. Never reuse generic layouts, names, branding or previous examples.
Never output placeholder years, fake testimonials, fake companies or incorrect branding.
Prefer automation over manual configuration. Detect things automatically whenever possible.
Reduce clicks. Save the user's time.
AI should understand intent instead of asking unnecessary questions.
All FounderLab tools must work together as one AI workspace, not isolated pages.
Chat can create Notes, Tasks, Code projects, Websites and YouTube workflows automatically when useful.
Notes must be intelligent, Tasks must be actionable, Builder must be production-ready, Code AI must produce clean maintainable code.
Generated code must be readable, modular, typed, documented and easy to extend.
Before completing any task, verify there are no broken links, broken buttons, console errors, layout issues or responsiveness problems.
Always optimise UX before adding more features.
Do not create unnecessary settings, switches or complexity.
Prefer clean design, smooth animations, fast performance and excellent accessibility.
Always preserve existing architecture unless a better implementation is clearly justified.
Never hardcode secrets or expose API keys.
When modifying a feature, build on top of the current implementation instead of replacing it.
Before finishing, ask yourself:
• Would someone pay for this?
• Does this feel premium?
• Would I proudly ship this?
If the answer is "no", continue improving until the answer becomes "yes".Never consider a task finished after writing code. Test the result against the original request. If any requested feature is missing, incomplete, outdated, broken or lower quality than requested, continue iterating until every requirement is satisfied.
